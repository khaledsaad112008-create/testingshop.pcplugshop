Last updated: 2026-07-14

project without re-discovering everything from scratch.

**This replaces the original `docs/PROJECT_CONTEXT.txt`** (deleted), which
described a static GitHub Pages + Cloudflare Worker architecture — this

## What this project is now

Originally: a static e-commerce site (plain HTML/CSS/JS) for a Qatar-based
PC hardware shop, hosted on GitHub Pages with a small Cloudflare Worker
handling two write endpoints.

**As of 2026-07-14: fully self-hosted.** The site now runs entirely from a
Node/Express server (`server/`) on the owner's personal Windows PC, exposed
to the internet via a Cloudflare Tunnel. GitHub Pages is no longer the live
host — DNS for `vivapcplug.dpdns.org` points directly at the tunnel. The
repo is still used as a backup/version-history layer (the server
auto-commits every data change), not as the runtime.

**Why**: the owner wanted to eliminate every manual step — no more
"Publish to GitHub" buttons, no localStorage drafts, no manual Excel
downloads. Admin edits now write directly to the server and are live
instantly; a monthly sales report is generated automatically with zero
human involvement.

**Important nuance**: this is explicitly described by the owner as "a test
on my personal computer" — they do **not** want it running 24/7 unattended.
Both services are configured for **manual start only** (see below), and the
owner turns the whole site on/off deliberately (e.g. off while gaming). The
live site going down is an accepted, intentional tradeoff here, not a bug.


## Architecture

```
Customer/Admin browser
        │  https://vivapcplug.dpdns.org  (Cloudflare DNS, proxied)
        ▼
Cloudflare edge — Access policy still gates /admin/* exactly as before
        │  (Cloudflare Tunnel — zero inbound ports opened on the home
        │   network, no dynamic-DNS needed, no direct origin to bypass
        │   Cloudflare Access through)
        ▼
cloudflared  →  http://localhost:3000
        ▼
Express server (server/index.js), on the owner's PC
  ├─ serves static site (css/js/images/html) from an explicit allowlist
  ├─ GET  /api/products                      (public, no Access needed)
  └─ /admin/api/*                            (Access-gated)
        ▼
data/products.json, data/sales.json, sales-reports/*.xlsx  (local disk,
source of truth) → auto `git commit && push` to `main` (fire-and-forget,
for history/backup — GitHub Pages hosting is left dormant, unused)
```

Both the server and the tunnel run as **Windows Services** (via NSSM), set
to **manual start** — they do NOT launch automatically on boot. See
"Turning the shop on/off" below.

## Route table

Public (no Cloudflare Access — customers must browse without OTP):
- `GET /api/products`

Admin (`/admin/api/*`, inherits the same Cloudflare Access gate that
protects `/admin/*` generally):
- `POST /admin/api/products`, `PUT /admin/api/products/:id`,
  `DELETE /admin/api/products/:id`
- `GET /admin/api/sales`, `POST /admin/api/sales`,
  `PUT /admin/api/sales/:id`, `DELETE /admin/api/sales/:id`
- `POST /admin/api/sales/export` `{monthKey}` — manual "regenerate this
  month's report now" (kept as an on-demand override, not a "download
  prompt" — no client-side generation or file dialog involved)
- `GET /admin/api/sales-reports/:month` — downloads an already-generated
  `.xlsx` straight off disk

Sales has no public read use case, so it all sits under `/admin/api/*`.


## Data & concurrency

`server/lib/dataStore.js`: reads are plain `fs.readFile` + `JSON.parse`
(no caching — always fresh off disk). Writes go through
temp-file-then-rename (atomic) plus an in-process promise-chained queue per
file path, so two near-simultaneous admin requests can't corrupt a file.

`.gitignore` has `*.xlsx` blanket-ignored, with an explicit exception:
```
*.xlsx
!sales-reports/*.xlsx
```
Without that exception, generated reports would silently never get staged
by git. (Two stray untracked `.xlsx` test-export files may still sit in
the repo root from earlier manual testing — harmless, ignored.)


## Monthly report automation

**Not** an exact-time scheduler (fragile against the PC being off/asleep —
miss that one hour window and the month is silently skipped forever).
Instead, `server/lib/cron.js` runs an **idempotent catch-up check**: on
server startup, and every ~30 minutes after, if today is on/after the 1st
of the month and `sales-reports/{previousMonthKey}.xlsx` doesn't already
exist, it generates one from `data/sales.json`. The file's own existence is
the idempotency marker — self-heals through any gap in uptime (which, given
this runs on a PC that gets turned off, happens often — this is exactly why
this design was chosen over exact-time firing).

Because the site is often OFF (manual-start, owner's choice), **a given
month's report may not get generated exactly on the 1st** — it'll appear
the next time the server happens to be running after that date. This is
expected behavior, not a bug, given the manual-start operating model.


## Git auto-sync

`server/lib/git.js`: after each successful write (product/sale CRUD,
monthly export), fire-and-forget `git add <specific paths> && git commit
&& git push` — never blocks the API response. Serialized (one git
operation at a time). Adds specific paths only, never `-A`. On a rejected
push, retries once with `git pull --rebase --autostash` then pushes again.
Logs and moves on on failure — the local file write already succeeded and
is the live source of truth regardless of whether the backup push landed.

**Two real bugs were found and fixed during testing** (worth knowing if
similar symptoms recur):
1. The `git()` exec wrapper wasn't capturing `stdout` on failure, only
   `stderr` — so a harmless "nothing to commit" case looked like a real
   failure.
2. The "no-op commit" detection regex only matched git's "nothing to
   commit, working tree clean" phrasing. When the wider working tree has
   *other* uncommitted changes (as it did throughout this migration), git
   instead says **"no changes added to commit"** for a no-op path-scoped
   commit — a different phrasing that wasn't originally handled.

**Unverified**: whether `git push` actually succeeds when the server runs
as a Windows Service (as opposed to an interactive shell, where it's
proven to work). Windows Services often run under a different security
context than the interactive user, which may not have access to the
cached Git Credential Manager login. Nobody has confirmed this either way
yet post-migration — if product/sales edits stop showing up as commits on
GitHub, this is the first thing to check.


## Turning the shop on/off

The owner explicitly wants manual control, not always-on — this is framed
as "a test" on a personal PC they also game on, and they don't want
background services affecting that (in practice the resource footprint is
trivial, but the *operational* preference is to control it deliberately).

Both `PCPlugServer` and `CloudflaredTunnel` are Windows Services set to
**`SERVICE_DEMAND_START`** (manual) — they do not launch on boot.

- **`scripts/start-shop.bat`** — double-click, accept one UAC prompt,
  starts both services.
- **`scripts/stop-shop.bat`** — double-click, accept one UAC prompt, stops
  both services. The site goes **fully offline** for everyone until
  started again.

Both scripts (`scripts/*.ps1` under the hood) self-elevate via UAC and
pause with "Press Enter to close" on completion or failure — earlier
versions closed instantly on a failed/declined elevation, showing red
error text for only a fraction of a second. If troubleshooting a start/stop
failure, that's the fix already in place; check the actual error text now
displayed instead.

Because start is manual, **sleep/hibernate/power settings were
deliberately left untouched** — if the PC sleeps, the site goes down the
same way stopping it manually would, which is consistent with the owner's
"I control when it's live" preference. Not worth "fixing" unless the owner
later wants the site to survive idle sleep during hours it's supposed to
stay live.


## Cloudflare Tunnel + service setup — what's actually installed

- Tunnel name: `pcplug`, ID `ecc38132-5c6c-47c7-b457-25d21997bd66`.
- Config: `C:\Users\khale\.cloudflared\config.yml` — ingress rule routing
  `vivapcplug.dpdns.org` → `http://localhost:3000`, catch-all 404.
- Credentials file: `C:\Users\khale\.cloudflared\ecc38132-...json`.
- DNS: a CNAME for `vivapcplug.dpdns.org` was added via
  `cloudflared tunnel route dns --overwrite-dns pcplug vivapcplug.dpdns.org`
  (the `--overwrite-dns` flag was required since a record already existed,
  pointing at GitHub Pages).

**Known gotcha, already worked around**: `cloudflared.exe service install`
(cloudflared's own native Windows service installer) silently ignores
whatever `--config` flag is passed to it — the registered service always
launches with zero arguments regardless, so it never actually connects to
anything useful, no error is thrown, `Get-Service` just shows "Running"
while doing nothing. **Workaround used**: wrap the tunnel with **NSSM**
instead (the same tool used for the Express server), passing the full
command explicitly via `AppParameters`:
```
--config "C:\Users\khale\.cloudflared\config.yml" tunnel run pcplug
```
If this service ever needs reinstalling, use NSSM with that exact
parameter string — do not use `cloudflared service install` directly, it
does not work as documented on this setup.

Also encountered: a Windows service can get stuck in `STOP_PENDING` if a
`Stop-Service` call is interrupted (e.g. the terminal window closed
mid-command). Fix: `Stop-Process -Id <pid> -Force` on the specific
underlying process (find via `tasklist`, look for the one in the
"Services" session — NOT any process in the "Console" session, which may
be a legitimate interactively-run instance), then `sc.exe delete
<ServiceName>`, then reinstall clean.


## The DNS cutover — how it was actually done

Two things had to happen in the right order (this matters if it's ever
redone or reverted):

1. **DNS first**: `cloudflared tunnel route dns` was run *before* removing
   the old Cloudflare Worker's routes. Reasoning: Cloudflare Workers Routes
   match by hostname+path at the edge *independent of what DNS points to*
   and take precedence over normal origin resolution. If the Worker's
   routes for `/admin/api/*` were removed *first*, while DNS still pointed
   at GitHub Pages (which was, at that moment, still serving the
   *pre-migration* frontend code that calls those exact old routes), the
   *currently-live* admin panel's Publish/Sync buttons would have broken
   immediately, for real. Flipping DNS to the tunnel first means the new
   (already-migrated) frontend takes over immediately, and only a narrow
   window of `/admin/api/sales/:id` (PUT/DELETE) and
   `/admin/api/sales/export` calls could have been mis-intercepted by the
   dying old Worker in the brief gap — much smaller blast radius.
2. **Then** the old Worker's routes were removed (`wrangler.toml` edited,
   `wrangler deploy` run) — closing that gap.
3. **Then** the migration code was committed and pushed to `main` (mostly
   for backup/history at that point, since the local server already had
   the correct code on disk and doesn't depend on git to run).

The Worker's Cron Trigger was deliberately left in place as a redundant
backup during the transition — see "Docs that need updating" below for
the full decommission step that hasn't happened yet.
