WHAT THIS PROJECT IS
---------------------
Static e-commerce site for a Qatar-based PC hardware shop (CPUs, GPUs, full
builds, peripherals). No frontend framework, no build tools — plain HTML,
CSS, and vanilla JS. There IS a small serverless backend now (a Cloudflare
Worker), used only for a couple of admin write actions — everything else is
static files served from GitHub Pages.

Live site:  https://vivapcplug.dpdns.org
Repo:       khaledsaad112008-create/testingshop.pcplugshop, branch `main` only
Hosting:    GitHub Pages (origin) behind Cloudflare (DNS proxy + Cloudflare
            Access gating /admin/* with email OTP)
Admin:      /admin/login.html — password is a SHA-256 hash in js/admin.js
            (never plaintext). Two independent locks: Cloudflare Access
            first, then this app-level password.


DEPLOYMENT RULES (IMPORTANT — READ BEFORE PUSHING ANYTHING)
-------------------------------------------------------------
1. Only `main` deploys. A branch left unmerged never goes live no matter
   what. Always confirm `git branch --show-current` is `main` before
   pushing, or merge into `main`.
2. Cache-bust every JS/CSS change. Cloudflare edge-caches .js/.css for 4h vs
   10min for HTML. Every <script src>/<link> tag uses a `?v=N` query param —
   bump it whenever that file's content changes, or visitors see stale code
   for hours. data/products.json and data/sales.json are fetched with
   `cache: "no-store"` and need no version bump.
3. Verify against origin, not the public URL, right after pushing:
   curl --resolve vivapcplug.dpdns.org:443:185.199.108.153 <url>
   This bypasses Cloudflare's cache to confirm GitHub's build actually
   finished before checking the public URL — hitting the public URL too
   early caches whatever stale content was there for another 4h.
4. Only commit/push when the user explicitly asks — but once asked, the
   workflow is "push straight to main," not PR-based.
5. The Cloudflare Worker (see below) is a SEPARATE deploy from the site.
   Pushing to `main` only updates the static site via GitHub Pages. Worker
   code changes need `npx wrangler deploy` run separately (from the repo
   root, using the already-authenticated `wrangler login` session).

HOW THE ADMIN "DATABASE" ACTUALLY WORKS (product catalog)
-------------------------------------------------------------
This was true before this session and still is for products.json:
1. Admin edits in the dashboard -> saved to localStorage
   ('pc_plug_products_v3'), which sets a dirty flag.
2. While dirty, that browser will NOT auto-refresh from the published
   data/products.json (protects the unsaved draft).
3. "Publish to GitHub" button -> POSTs to the Worker
   (/admin/api/publish) -> Worker commits data/products.json to `main`
   directly via GitHub's Contents API using a secret GITHUB_TOKEN.
4. "Reload from Database" discards the local draft and re-fetches the
   published file.

docs/BACKEND.md says "No backend exists" — that was true when it was
written, but is NO LONGER fully accurate. There is now a small Cloudflare
Worker (see worker/ and wrangler.toml) that:
  - Commits data/products.json to GitHub on publish (pre-existing, not

  - Commits data/sales.json to GitHub on sync (NEW, this session)
  - Runs on a monthly Cron Trigger to auto-generate
    sales-reports/YYYY-MM.xlsx (NEW, this session)
  - Also exposes a manual "force export" endpoint for the same report
    generation, callable on demand from the admin UI (NEW, this session)

accounts, no database beyond these two JSON files in the repo. The
Cloudflare Access + password gate on /admin/* is still the real security
boundary; the Worker's routes themselves don't independently authenticate
requests (same trust model as the original publish endpoint).

touched yet.


THE SALES FEATURE (built this session — full detail)
-------------------------------------------------------------
User's original ask: a flexible way to log sold products — including ones
never added to the catalog, or sold at a price that only applied for one
month — grouped by month, auto-saved to Excel every month both locally and
in the GitHub repo, with a trading summary visible on the admin homepage.

Data model (a "sale" record, see js/sales.js):
{
  id, date (YYYY-MM-DD), month (YYYY-MM, derived from date),
  productId (nullable — null means not linked to a real catalog product),
  name, category, quantity, unitPrice, total, note, loggedAt (timestamp)
}
Price/name/category are ALWAYS freely editable at log time, regardless of
whether the item matches a real catalog product — this is what makes it
flexible enough to log off-catalog items or a price that only held for one
month.

Admin UI — admin/sales.html (new nav tab "Sales" on all admin pages):
  - "Log a Sale" form: product name field has a <datalist> of existing
    catalog product names for autofill (name+category+price), but you can
    type anything — it's a free-text field, not a strict dropdown.
  - "This Month's Sales" table: current month's entries, inline edit/delete
    (same UI pattern as the product table in dashboard.html).
  - "Monthly Archive" table: every past month that has sales, with
    per-month revenue/units/count and two actions per row:
      "Download .xlsx"        — local-only download, no network call
      "Sync & Export"         — syncs local draft to GitHub, then forces
                                 that month's sales-reports/*.xlsx to update
                                 on GitHub right now
  - Top-level buttons above "This Month's Sales":
      "Download This Month (.xlsx)" — local download of current month
      "Sync to GitHub"               — commits data/sales.json only
      "Sync & Export to GitHub Now"  — sync + force-export current month

Month rollover (the "starts fresh with the other month" requirement):
  - There's no explicit "close the month" action needed — every sale is
    tagged with its month at creation time, so grouping/archiving is
    automatic based on that tag.
  - js/sales.js checkSalesMonthRollover() runs on every Sales page load. It
    compares the wall-clock month to the last month this browser saw
    (localStorage key pc_plug_sales_last_seen_month). If the month changed,
    it auto-downloads that just-closed month's .xlsx locally right there in
    the browser (a real client-side file download, not just a log message).
  - This only fires from an actual browser visit — it is NOT how the
    GitHub-side automatic report happens (that's the cron trigger, below).
    Two separate, independent auto-export mechanisms by design, per the
    user's explicit request for "both locally and in github."

Automatic monthly export to GitHub (the cron trigger):
  - worker/wrangler.toml has `[triggers] crons = ["0 1 1 * *"]` — runs at
    01:00 UTC on the 1st of every month.
  - worker/sales.js `handleMonthlyExport()` (called from the Worker's
    `scheduled()` export in worker/index.js) reads data/sales.json AS
    CURRENTLY PUBLISHED ON GITHUB, filters to the previous calendar month,
    builds an .xlsx with SheetJS (`xlsx` npm package), and commits it to
    sales-reports/YYYY-MM.xlsx via GitHub's Contents API.
  - CRITICAL CAVEAT: this only sees sales that were actually synced to
    GitHub via the "Sync to GitHub" / "Sync & Export" buttons at some point
    during the month. Sales sitting only in a browser's localStorage that
    were never synced will NOT appear in the automatic report. The user has
    been told this — worth reminding them again if sales reports look thin.
  - If there are zero sales for that month in the synced data, the handler
    returns `{ skipped: true }` and writes nothing (no empty report files
    clutter the repo).

Manual force-export (the safety net, added after initial build):
  - Endpoint: POST /admin/api/sales/export, body { monthKey: "YYYY-MM" }
  - Calls the same underlying exportMonthReport() logic the cron uses, just
    parameterized by month instead of always using "previous month."
  - Wired to two buttons in admin/sales.html (see above). Both buttons sync
    first, then export, so the report always reflects the latest local
    edits — no separate manual sync step needed before using them.
  - This was added specifically because Cron Triggers only fire once a
    month, so there was no way to verify the automatic export actually
    works end-to-end without waiting for the real date (or writing a
    disruptive test commit to production, which was avoided). It also
    gives the admin a way to regenerate a report if they correct a sale
    after the month has already closed.

Dashboard homepage (admin/dashboard.html):
  - New "Trading — <Month>" panel above "Add New Product", showing current
    month's Revenue / Units Sold / Sales Logged, reading from the same
    localStorage sales data. Links to the Sales page.


CLOUDFLARE WORKER — INFRASTRUCTURE NOTES
-------------------------------------------------------------
- Worker name: pcplug-publish-worker (name kept from before this session,
  even though it now does more than just "publish" — not renamed to avoid
  unnecessary churn on a working deployment).
- Routes (see wrangler.toml):
    vivapcplug.dpdns.org/admin/api/publish*
    vivapcplug.dpdns.org/admin/api/sales/*
- Secret: GITHUB_TOKEN (set via `wrangler secret put GITHUB_TOKEN`, not in
  any file in this repo). Used for all GitHub Contents API calls (read sha,
  write/commit files) across publish.js and sales.js.
- Deploy command: `npx wrangler deploy` from repo root. Requires being
  logged in via `npx wrangler login` (already done once this session, tied
  to khaledsaad112008@gmail.com's Cloudflare account).
- KNOWN QUIRK: the wrangler CLI's OAuth token (from `wrangler login`) does
  NOT have permission to manage Cron Triggers via the API — attempting to
  set `[triggers] crons = [...]` via `wrangler deploy` returns a 403 on the
  `/workers/scripts/{name}/schedules` endpoint. Re-running `wrangler login`
  does NOT fix this — it's not a stale-token issue, the scope just isn't
  granted to CLI OAuth sessions. WORKAROUND USED: the cron trigger was added
  manually via the Cloudflare dashboard (Workers & Pages -> the worker ->
  Triggers -> Cron Triggers -> add "0 1 1 * *"). Once added there, it
  persists correctly and subsequent `wrangler deploy` runs do NOT clobber
  it (confirmed — a later deploy in this session showed
  "schedule: 0 1 1 * *" as successfully reconciled, no error). So: routine
  code deploys are safe and won't remove the cron trigger, but if the cron
  trigger ever needs to be changed or re-added from scratch, it has to be
  done via the dashboard UI, not wrangler.
- `xlsx` (SheetJS) dependency: installed from SheetJS's own CDN tarball
  (https://cdn.sheetjs.com/xlsx-0.20.3/xlsx-0.20.3.tgz), NOT from the plain
  npm registry — the npm registry's `xlsx` package is stuck at an old,
  vulnerable, unmaintained 0.18.5. If this ever needs updating or
  reinstalling, use the same CDN-tarball approach, not `npm install xlsx`.
